## 作业(1)：Mutex — 持有线程未释放就结束时，等待方会怎样？

### 作业(1) 题目来源

`Assignment4/01 线程同步/01 Mutex/Program.cs` 文末注释：

> 编程说明如果拥有互斥体的线程没有释放互斥而结束，其他等待线程是否会死等?

### 结论（.NET / Windows 行为概要）

- **一般不会无限死等**：当**拥有互斥体的线程终止且未调用 `ReleaseMutex`** 时，操作系统会把该互斥体标记为**被抛弃（abandoned）**。
- **下一个成功等到该互斥体的线程**在 .NET 中通常会收到 `**AbandonedMutexException`**，用于提醒：“你拿到的锁来自异常终止的上家，共享不变量可能已损坏。”
- 因此：**不是**简单地在 `WaitOne` 里永远阻塞到地老天荒；但也不是“毫无代价地安全”——需要捕获异常并决定是记录日志后 `ReleaseMutex` 还是让进程失败。

### 与 `lock` 的对比（概念）

- `lock` 是编译器配合 `Monitor` 的**进程内**构造，线程退出时不会留下“可被另一线程 `Wait` 到的内核 Mutex 句柄”这种同一模型；本题讨论的是 `**System.Threading.Mutex`** 这种可跨进程的同步对象。

### 可运行示例

见 `01_Mutex作业1/`，运行后观察等待线程是否收到 `AbandonedMutexException`。

### 参考（自行查阅官方文档）

- `Mutex` / `AbandonedMutexException`（Microsoft Learn）

## 作业2：设计实验说明 `EnterUpgradeableReadLock` 的优势

### 作业2 题目来源

`Assignment4/01 线程同步/07 ReaderWriterLockSlim/Program.cs` 文末注释：

> 设计一个实验说明 EnterUpgradeableReadLock 的优势

### 实验思路（摘要）

- **背景**：写线程常需要先**读检查**（例如 `ContainsKey`）再**可能写入**。若从头到尾使用 **写锁**，读线程会被完全挡在门外。
- **可升级读锁**：同一线程可先进入 **可升级读锁**，在“只读检查 + 非关键区工作”阶段与**普通读锁读者**更好地并发（仍受“同时仅一个可升级读”等规则约束）；仅在确需修改时再 **升级为写锁**，缩短写锁占用时间。
- **本仓库 runnable**：`02_RWLock作业2/Program.cs` 用两种策略跑同一负载，对比在固定时间窗内**读者完成读次数**（写者频率固定），数值高者说明读并发更好。


### 作业2 扩展思考（可自行补充到实验报告）

- 若写比例极高，可升级读锁也会成为瓶颈（任一时刻仅一个可升级读者），需结合真实访问模式测量。

## 作业4：查找资料，熟悉以下并发集合用法

### 作业4 题目来源

`Assignment4/03 并发集合/01 ConcurrentDictionary/Program.cs` 与 `01 ConcurrentDictionaryTest/Program.cs` 文末注释：

> 查找资料，熟悉以下数据结构的用法  
> (1) ConcurrentDictionary (2) ConcurrentQueue (3) ConcurrentStack (4) ConcurrentBag (5) BlockingCollection

以下笔记为课程级摘要，详细 API 以 [Microsoft Learn — System.Collections.Concurrent](https://learn.microsoft.com/dotnet/api/system.collections.concurrent) 为准。

### (1) ConcurrentDictionary<TKey, TValue>

- **用途**：线程安全的键值表；多线程下可并发读写，无需外层 `lock`。
- **常见操作**：索引器 `dict[key] = v`（添加或更新）、`TryAdd`、`TryRemove`、`TryGetValue`、`AddOrUpdate`、`GetOrAdd`。
- **注意**：枚举（`foreach`）与快照语义与 `Dictionary` 不同；高并发下注意选择合适的更新 API 以避免多余分配。

### (2) ConcurrentQueue

- **用途**：FIFO 队列，多生产者多消费者安全。
- **常见操作**：`Enqueue`、`TryDequeue`、`TryPeek`。
- **注意**：无阻塞 dequeue；若需要“队列为空则等待”，配合 `BlockingCollection` 或自旋/事件。

### (3) ConcurrentStack

- **用途**：LIFO 栈，多线程安全。
- **常见操作**：`Push`、`TryPop`、`TryPeek`。
- **注意**：与 `ConcurrentQueue` 类似，为非阻塞原语。

### (4) ConcurrentBag

- **用途**：无序元素袋；**优化多线程各自添加/取用自己添加的元素**时的性能（线程局部性）；遍历顺序不确定。
- **常见操作**：`Add`、`TryTake`。
- **注意**：不适合需要严格全局 FIFO/LIFO 语义的场景；`TryTake` 取出的元素顺序不保证。

### (5) BlockingCollection

- **用途**：在并发集合之上提供 **上限容量** 与 **阻塞** 语义的生产者-消费者边界。
- **常见用法**：内部可包装 `ConcurrentQueue` 等；`Add`/`Take` 在满/空时阻塞；`CompleteAdding` 通知消费者结束。
- **注意**：实现 `IDisposable`，用完应释放；适合管道、任务队列、有界缓冲。
